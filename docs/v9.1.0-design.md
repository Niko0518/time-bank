## 1. 背景与问题陈述

### 1.1 v9.0.0 架构哲学回顾

v9.0.0 引入**服务端权威写入架构**：

| 维度 | 旧架构 | v9.0.0 架构 |
|------|--------|-------------|
| 写入方式 | 客户端 `db.collection().add/update()` | `callMutation()` → 云函数写入 |
| 原子性 | 客户端多写可能漂移 | 云函数 `_.inc()` 原子更新 |
| 回滚 | 无 | onRollback + 失败队列 |
| Watch 处理 | 5 种分支 | 3 种统一 |

v9.0.0 解决了**单条 mutation 的原子性**——一条 mutation 要么全成功，要么全失败（云函数事务边界内）。

### 1.2 v9.0.0 未解决的层

v9.0.0 完善了"单 mutation 机制"，但**"复合 mutation 机制"这一层是缺失的**。

业务层经常需要把多个 mutation 组合为一个逻辑事务：

| 业务路径 | 涉及 mutation | 是否原子？ |
|----------|----------------|------------|
| **完成计时任务** | addTransaction + stopTask | ❌ 否（v9.0.5 现状） |
| **完成习惯任务** | addTransaction + stopTask + (可选) updateHabitStreak | ❌ 否 |
| **删除任务（带交易）** | deleteTransaction + deleteTask | ❌ 否 |
| **撤销交易** | deleteTransaction | ✅ 是（单 mutation 原子） |
| **屏幕时间补录** | addTransaction + updateDailyChange | ❌ 否 |
| **跨设备冲突解决** | updateTransaction（覆盖） | ✅ 是 |

### 1.3 真实案例：P0-B 现象

**用户反馈**（v9.0.5 评估期间）：

> "在安卓端使用时，某计时任务点击结束，立即计入交易记录、改变余额，但任务在瞬间复活，但这次没有撤回交易和余额变化，仅仅是任务复活。同时，失败队列报错数据不存在stoptask"

**根因分析**：

```
用户点击结束任务
    │
    ├─ app-2.js:4820 runningTasks.delete(taskId)         ← 本地删除
    ├─ DAL.stopTask(taskId)
    │     └─ callMutation('stopTask', ...)               ← mutation A
    ├─ processNormalCompletion
    │     └─ addTransaction(transaction)
    │           └─ callMutation('addTransaction', ...)   ← mutation B
    │
    ├─ 场景：mutation A (stopTask) 云端 1003
    │     ├─ onRollback 触发 → runningTasks 复活
    │     └─ mutation B (addTransaction) 已成功
    │
    └─ 状态：交易存在 + 余额已变 + 任务复活（数据不一致）
```

**v9.0.5 修复**（P0-A）：修正 onRollback 快照使用 taskData（不是 _id）——**修复了任务复活的"数据完整性"**。

**未修复**（P0-B）：**任务仍然会"复活"，addTransaction 仍然不回滚**——根本问题没解决。

### 1.4 "打补丁"vs"机制完善"

继续"打补丁"会怎样？

| 业务路径 | 需要的"联动" |
|----------|----------------|
| `completeTask` | addTx 成功 + stopTask 失败 → 补偿 addTx |
| `completeHabit` | addTx 成功 + stopTask 失败 → 补偿 addTx + 还原 habitStreak |
| `completeSleep` | addSleep 成功 + stopTask 失败 → 补偿 addSleep |
| `deleteTask` | deleteTask 成功 + deleteTx 失败 → 还原任务 |

**N 个业务路径 × N 个补偿点 = 复杂度爆炸**——这正是要避免的"打补丁地狱"。

**机制完善**：**新增"复合 mutation"层，让所有"组合操作"自动获得原子性**。

## 2. 设计目标与原则

### 2.1 设计目标

| # | 目标 | 衡量指标 |
|---|------|----------|
| G1 | **多 mutation 组合的原子性保证** | 复合操作要么全成功，要么全回滚（云端权威） |
| G2 | **业务层零手工联动** | 业务层只需声明"哪些步骤组合"，无需手工写补偿逻辑 |
| G3 | **失败时单一原子语义** | 业务层只需一个 onRollback，处理"复合操作整体失败" |
| G4 | **现有 v9.0.0 哲学延续** | 不破坏"服务端权威 + 乐观更新 + 失败队列" |
| G5 | **向前兼容** | 旧客户端不调用 compound action，云端正常服务 |

### 2.2 设计原则

延续 v9.0.0 以来"机制完善"哲学：

1. **根因导向**：每个版本问"为什么要打这个补丁"——根因 = 缺失的机制层
2. **一次解决一类**：不修单个 bug，修整个机制
3. **业务层不手工联动**：业务层代码应该"声明意图"，机制自动保证正确性
4. **服务端权威**：原子性保证在云端，客户端信任云端结果
5. **失败不静默**：复合操作失败时，错误码 + 通知 + 队列化（v9.0.2 哲学）

## 3. 架构概览

### 3.1 新增概念：Compound Mutation

**定义**：Compound Mutation 是**云端原子执行**的多个 mutation 序列，对外表现为单一原子单元。

**类比**：
- 数据库事务（Transaction）：多个 SQL 在事务边界内原子执行
- 分布式 Saga：多个子事务 + 补偿机制
- 事件溯源（Event Sourcing）：多个事件作为一个聚合根的命令

### 3.2 整体架构

```
┌─────────────────────────────────────────────────────────┐
│ 业务层（如 completeTask）                                 │
│   ↓                                                      │
│ callCompoundMutation([step1, step2, step3], options)     │
│   ↓                                                      │
│ 客户端 callMutation (action='compound', data={steps})   │
│   ↓                                                      │
│ 云函数 tbMutation                                        │
│   ├─ case 'compound':                                    │
│   │     ├─ 预检所有步骤                                   │
│   │     ├─ 顺序执行（executeAction 递归）                  │
│   │     ├─ 任一失败 → rollbackCompletedSteps             │
│   │     └─ 全部成功 → 返回 code: 0                       │
│   ↓                                                      │
│ 客户端                                                  │
│   ├─ 成功 → onSuccess 触发（业务层无需操作）               │
│   └─ 失败 → onRollback 触发（业务层做"对外可见的回滚"）    │
└─────────────────────────────────────────────────────────┘
```

### 3.3 关键设计决策

| 决策点 | 选 A | 选 B | 选 C | **最终选择** |
|--------|------|------|------|--------------|
| 复合粒度 | action 级 | 数据级 | 调用级 | **action 级**（与 v9.0.0 一致） |
| 失败补偿 | 云端自动 | 客户端 | 混合 | **云端自动**（保证原子性） |
| onRollback 触发 | 业务错误 + 网络 | 业务错误 | 网络 | **业务错误**（与 v9.0.2 一致） |
| 步骤数上限 | 无 | 5 | 10 | **10**（防止滥用） |
| 步骤嵌套 | 允许 | 禁止 | 允许但限 1 层 | **禁止**（防止指数爆炸） |

## 4. 云端设计

### 4.1 通用执行框架

`tbMutation` 重构：所有 action 通过 `executeAction(action, data, context)` 递归执行。

```javascript
// tbMutation/index.js
async function executeAction(action, data, context) {
    const { _openid: uid } = context;
    switch (action) {
        case 'addTransaction':    return addTransaction(data, context);
        case 'updateTransaction': return updateTransaction(data, context);
        case 'deleteTransaction': return deleteTransaction(data, context);
        // ... 其他 action
        default: return { code: 400, message: `未知 action: ${action}` };
    }
}
```

### 4.2 compound action 实现

```javascript
// tbMutation/index.js
const MAX_COMPOUND_STEPS = 10;

case 'compound': {
    const { steps = [] } = data;
    
    // 1. 预检：步骤数、参数
    if (steps.length === 0) {
        return { code: 400, message: '复合操作无步骤' };
    }
    if (steps.length > MAX_COMPOUND_STEPS) {
        return { code: 400, message: `复合操作步骤数超限（${MAX_COMPOUND_STEPS}）` };
    }
    for (let i = 0; i < steps.length; i++) {
        if (!steps[i] || !steps[i].action || !steps[i].data) {
            return { code: 400, message: `步骤 ${i+1} 格式错误` };
        }
    }
    
    // 2. 顺序执行
    const results = [];
    const completedSteps = [];  // { step, result, contextSnapshot }
    const context = { _openid: uid };
    
    try {
        for (let i = 0; i < steps.length; i++) {
            const step = steps[i];
            const stepResult = await executeAction(step.action, step.data, context);
            
            // 3. 成功或幂等：记录到 completedSteps
            if (stepResult.code === 0 || stepResult.code === 410) {
                results.push(stepResult);
                completedSteps.push({ step, result: stepResult, context });
                continue;
            }
            
            // 4. 业务失败：回滚已完成步骤
            await rollbackCompletedSteps(completedSteps, context);
            return {
                code: 1001,
                message: `复合操作第 ${i+1} 步失败：${stepResult.message}`,
                failedStepIndex: i,
                rolledBackSteps: completedSteps.length
            };
        }
        
        return { code: 0, message: '复合操作全部成功', results };
    } catch (e) {
        // 5. 异常：回滚 + 抛出
        await rollbackCompletedSteps(completedSteps, context);
        console.error('[tbMutation.compound] 异常:', e);
        return { code: 500, message: '复合操作异常：' + e.message };
    }
}
```

### 4.3 rollbackCompletedSteps 实现

每个 action 都有对应的反向操作：

```javascript
async function rollbackCompletedSteps(completedSteps, context) {
    console.log(`[rollback] 开始回滚 ${completedSteps.length} 步`);
    
    // 逆序回滚（后执行的先回滚）
    for (let i = completedSteps.length - 1; i >= 0; i--) {
        const { step, result } = completedSteps[i];
        try {
            await rollbackAction(step.action, step.data, result, context);
            console.log(`[rollback] 步骤 ${i+1} (${step.action}) 回滚成功`);
        } catch (e) {
            console.error(`[rollback] 步骤 ${i+1} (${step.action}) 回滚失败:`, e);
            // 回滚失败不抛出（避免覆盖原始错误）
            // 但记录到告警日志，运维介入
        }
    }
}

async function rollbackAction(action, data, result, context) {
    switch (action) {
        case 'addTransaction':
            // 反向：删除刚加的交易
            return deleteTransaction({ ...data, txId: result.txId || data.txId }, context);
        case 'updateTransaction':
            // 反向：恢复旧值
            return updateTransaction({ ...data, _rollback: true, _originalData: data.prevTx }, context);
        case 'deleteTransaction':
            // 反向：重新加回
            return addTransaction(data._originalTx, context);
        case 'startTask':
            // 反向：停止
            return stopTask({ taskId: data.taskId }, context);
        case 'stopTask':
            // 反向：重新开始（如果原数据可恢复）
            return startTask(data._originalRunning, context);
        case 'saveTask':
            // 反向：恢复旧值
            return saveTask({ ...data, _rollback: true, _originalData: data.prevTask }, context);
        case 'deleteTask':
            // 反向：重新加回
            return saveTask(data._originalTask, context);
        case 'saveProfile':
            // 反向：恢复旧值
            return saveProfile({ ...data, _rollback: true, _originalData: data.prevProfile }, context);
        case 'updateDailyChange':
            // 反向：负向增量
            return updateDailyChange({ ...data, _rollback: true, delta: -data.delta }, context);
        case 'updateCachedBalance':
            // 反向：负向增量
            return updateCachedBalance({ ...data, _rollback: true, delta: -data.delta }, context);
        default:
            console.warn(`[rollback] 未实现 ${action} 的反向操作`);
            return { code: 0, message: 'no-op' };
    }
}
```

### 4.4 幂等性保证

复合操作必须**整体幂等**——重试时不会重复执行。

**设计**：
- 整个 compound 调用有一个 `mutationId`（v9.0.2 引入）
- 云端**首次**执行后，缓存 `mutationId → 已完成结果`（内存或 redis）
- 第二次相同 `mutationId` 调用，**直接返回缓存结果**

**实现**：

```javascript
const compoundResultCache = new Map();  // mutationId → result
const COMPOUND_CACHE_TTL = 24 * 60 * 60 * 1000;  // 24 小时

case 'compound': {
    const mutationId = event.mutationId;
    
    // 幂等检查
    if (mutationId && compoundResultCache.has(mutationId)) {
        console.log(`[tbMutation.compound] 幂等返回: ${mutationId}`);
        return compoundResultCache.get(mutationId);
    }
    
    // ... 执行复合操作 ...
    
    // 缓存结果
    if (mutationId) {
        compoundResultCache.set(mutationId, result);
        setTimeout(() => compoundResultCache.delete(mutationId), COMPOUND_CACHE_TTL);
    }
    
    return result;
}
```

## 5. 客户端设计

### 5.1 客户端包装函数

```javascript
// app-1.js
async function callCompoundMutation(steps, { onRollback } = {}) {
    const currentUid = await DAL.getCurrentUid();
    if (!currentUid) {
        throw new Error('未登录');
    }
    
    // 复用 callMutation 的失败处理、失败队列、通知机制
    return callMutation('compound', {
        _openid: currentUid,
        steps: steps.map(step => ({
            action: step.action,
            data: step.data
        }))
    }, { onRollback });
}
```

### 5.2 业务层使用

**v9.1.0 之前**（completeTask）：

```javascript
// app-2.js
async function completeTask(taskId, transaction) {
    // 两个独立 callMutation
    await DAL.addTransaction(transaction);
    await DAL.stopTask(taskId, runningTask);
}
```

**v9.1.0 之后**（completeTask）：

```javascript
// app-2.js
async function completeTask(taskId, transaction) {
    // 单一 callCompoundMutation
    await callCompoundMutation([
        { action: 'addTransaction', data: { ...transaction } },
        { action: 'stopTask', data: { taskId } }
    ], {
        onRollback: () => {
            // 复合操作整体失败，业务层做"对外可见的回滚"
            // （云端已完成反向操作，本地只需撤销乐观 UI）
            transactions = transactions.filter(t => t.id !== transaction.id);
            // 修正余额
            if (typeof currentBalance === 'number') {
                const delta = transaction.type === 'earn' ? -transaction.amount : transaction.amount;
                currentBalance += delta;
            }
            // 恢复 runningTasks（如果 addTransaction 阶段已经停了 stopTask 之前的乐观状态）
            // ... 视具体业务
            if (typeof updateAllUI === 'function') updateAllUI();
        }
    });
}
```

### 5.3 关键设计：业务层 onRollback 的语义

**语义转变**：

| v9.0.x 时代 | v9.1.0 时代 |
|--------------|--------------|
| 每个 mutation 各自有 onRollback | 复合 mutation 整体一个 onRollback |
| 业务层手工联动 | 业务层只需"整体失败时回滚" |
| 多个 onRollback 可能时序错乱 | 一个 onRollback，原子语义 |

**业务层 onRollback 的职责**：
- 撤销**本地乐观 UI**（云端已撤销服务端状态）
- 通知用户（如有必要）
- 触发 Watch 重新同步（如需要）

## 6. 业务迁移路径

### 6.1 迁移优先级

| 优先级 | 业务路径 | 当前调用 | 迁移后 |
|--------|----------|----------|--------|
| **P0** | `completeTask`（计时任务结束） | 2 callMutation | 1 callCompoundMutation |
| **P0** | `completeHabit`（习惯任务结束） | 2-3 callMutation | 1 callCompoundMutation |
| **P1** | `completeSleep`（睡眠记录） | 2 callMutation | 1 callCompoundMutation |
| **P1** | `screenTimeRecord`（屏幕时间补录） | 2 callMutation | 1 callCompoundMutation |
| **P2** | `deleteTaskWithTransactions`（删除任务带交易） | 2 callMutation | 1 callCompoundMutation |
| **P2** | `rebuildHabitStreak`（习惯重建） | 1-2 callMutation | 1 callCompoundMutation |

### 6.2 迁移步骤

每个业务路径迁移步骤：

1. **识别**当前是哪些 callMutation
2. **组合**为 steps 数组
3. **替换**为 callCompoundMutation
4. **设计** onRollback 逻辑（撤销本地乐观 UI）
5. **测试**正常 + 失败场景
6. **删除**原来的"联动"代码

### 6.3 兼容策略

- **旧客户端**：v9.1.0 之前不使用 compound action
- **新云端**：同时支持旧 action 和 compound action
- **新客户端**：默认用 compound action
- **回滚兼容**：如发现 v9.1.0 云端有 bug，可临时回退到旧路径（业务层双路径支持）

```javascript
// 业务层示例：双路径
async function completeTask(taskId, transaction) {
    if (USE_COMPOUND_MUTATION) {
        await callCompoundMutation([...]);
    } else {
        await DAL.addTransaction(transaction);
        await DAL.stopTask(taskId, runningTask);
    }
}
```

## 7. 错误码与错误处理

### 7.1 新增错误码

| Code | 含义 | 客户端处理 | 是否重试 |
|------|------|-----------|---------|
| 1005 | 复合操作步骤数超限 | onRollback + 通知 | ❌ |
| 1006 | 复合操作步骤格式错误 | onRollback + 通知 | ❌ |
| 1007 | 复合操作部分回滚失败 | onRollback + **告警通知** + 运维介入 | ❌ |
| 1008 | 复合操作步骤不支持 | onRollback + 通知 | ❌ |

### 7.2 与 v9.0.2 错误码体系一致

- 业务错误（1001-1008）：onRollback + 通知 + 不入重试队列
- 网络/限流/内部错误（429/500/503）：onRollback + 入队 + 重试
- 成功（0/410）：正常

### 7.3 失败队列扩展

`MutationFailureHandler` 需扩展：
- 复合操作的失败记录包含 `steps` 字段（哪些步骤）
- 重试时整个复合操作**整体重试**（不重试部分步骤）

```javascript
// MutationFailureHandler.recordFailure 扩展
recordFailure(mutation, error, stage) {
    const failure = {
        mutationId: mutation.mutationId,
        action: mutation.action,
        data: mutation.data,
        steps: mutation.data.steps,  // 新增
        error: error,
        stage: stage,
        timestamp: Date.now()
    };
    // ... 保存到 localStorage
}
```

## 8. 测试计划

### 8.1 单元测试（云端）

测试每个 action 的反向操作：

| 测试用例 | 验证点 |
|----------|--------|
| addTransaction → rollback | 交易被删除，余额还原，daily 还原 |
| startTask → rollback (stopTask) | running 文档被删除 |
| stopTask → rollback (startTask) | running 文档重新加回 |

### 8.2 集成测试（云端 + 客户端）

测试完整复合操作：

| 测试场景 | 步骤 | 期望结果 |
|----------|------|----------|
| 正常完成 | addTx + stopTask | 全成功，云端状态一致 |
| addTx 失败 | addTx 失败 + stopTask | stopTask 不执行，云端无变化 |

## 11. v9.1.0 范围扩展：启用 timebankTaskLock

### 11.1 为什么 v9.1.0 要同时启用任务锁

**`timebankTaskLock` 自 v7.30.0 引入但从未启用**：
- 云函数：60s TTL 锁，cache 存储
- 客户端：`DAL.lockTask` / `DAL.unlockTask` / `DAL.checkTaskLock` 已实现
- **业务层 0 处调用**

**为什么 v9.1.0 必须同时启用**：

| 单独启用 compound 不带任务锁 | 单独启用任务锁不带 compound |
|-------------------------------|-------------------------------|
| ❌ 两台设备同时复合操作 → 云端并发写，仍可能漂移 | ❌ 单 mutation 失败时无原子性 |
| ❌ 复合操作可能与单 mutation 冲突 | ❌ 跨设备同时 startTask + stopTask → 云端 update 覆盖 |
| ✅ **compound + taskLock 组合**：互斥 + 原子 | — |

**结论**：v9.1.0 必须**同时**引入 compound + 任务锁，单独任何一个都不完整。

### 11.2 任务锁的细粒度设计

**当前实现**（v7.30.0）：
- 锁粒度：任务级（taskId 维度）
- 持锁时长：60s（TTL）
- 申请：业务层调用 `DAL.lockTask(taskId)`

**v9.1.0 增强**：
- 锁粒度：保持任务级（足够）
- **复合操作整体持锁**（compound 入口申请，try/finally 释放）
- **细粒度步骤互斥**：compound 内部 steps 不需要各自持锁（已经被整体锁保护）

```javascript
// callCompoundMutation 内部包装任务锁
async function callCompoundMutation(steps, options = {}) {
    const taskIds = extractTaskIds(steps);
    const sortedIds = [...new Set(taskIds)].sort();  // 排序避免死锁
    
    // 1. 申请所有任务锁
    const locks = [];
    for (const taskId of sortedIds) {
        const lock = await DAL.lockTask(taskId);
        if (lock.code === 409) {
            // 其他设备正在操作
            for (const l of locks) await DAL.unlockTask(l.taskId);
            return { code: 409, message: lock.message, lockedBy: lock.lockedBy };
        }
        locks.push({ taskId });
    }
    
    try {
        // 2. 复合操作（云端原子）
        return await callMutation('compound', { steps }, options);
    } finally {
        // 3. 释放所有锁（finally 保证异常也释放）
        for (const l of locks.reverse()) {
            await DAL.unlockTask(l.taskId).catch(e => 
                console.warn(`[unlockTask] ${l.taskId} 释放失败（可能已自动过期）:`, e)
            );
        }
    }
}
```

### 11.3 业务层接入清单

v9.1.0 需要在以下业务路径接入任务锁：

| 业务路径 | 是否需要锁 | 理由 |
|----------|------------|------|
| `completeTask`（计时任务结束） | ✅ 需要 | 复合 addTx + stopTask |
| `completeHabit`（习惯任务结束） | ✅ 需要 | 复合 addTx + stopTask + habitStreak |
| `startTask` | ✅ 需要 | 防两设备同时 start 同一任务 |
| `stopTask` | ✅ 需要 | 防两设备同时 stop 同一任务 |
| `updateRunningTask`（暂停/恢复） | ✅ 需要 | 防暂停/恢复竞态 |
| `saveTask` | ✅ 需要 | 防编辑冲突 |
| `deleteTask` | ✅ 需要 | 防删除 + 完成的竞态 |
| `addTransaction`（手动添加） | ❌ 不需要 | 幂等 + 不影响其他设备状态 |
| `updateTransaction` | ❌ 不需要 | 幂等 |
| `deleteTransaction` | ❌ 不需要 | 幂等 |

### 11.4 409 错误的业务处理

任务锁被占用的 UX 设计：

```javascript
// 业务层统一处理 409
async function handleMutationResult(res, taskId) {
    if (res.code === 409) {
        // 其他设备正在操作
        showToast({
            type: 'warning',
            message: `任务正被 ${res.lockedBy} 操作`,
            duration: 3000
        });
        // 可选：自动重试（轮询 checkLock）
        return retryWithBackoff(() => callCompoundMutation(...), { maxRetries: 3 });
    }
    return res;
}
```

### 11.5 与 v9.0.0 错误的兼容

- v9.0.0 错误码：1001-1004（业务）+ 429/500/503（系统）
- v9.1.0 新增：**409（任务锁占用）**——客户端视为业务错误，不入重试队列
- 失败队列：409 错误**不入队**（用户决策而非自动重试）

|--------|------|------|----------|--------|----------|

| P1 | v9.3.0 | **离线完整化** | 离线时复合 + 状态机 + 锁的协作 | 9 天 | 依赖 v9.1.0 + v9.2.0 |

### 12.3 详细说明

#### v9.2.0 状态机一致性

**核心问题**：任务的"开始→暂停→恢复→停止"是状态机，但当前每个状态转换都是独立 mutation，**没有合法性检查**。

**典型场景**：
- 设备 A 点"暂停"，同时设备 B 点"恢复" → 云端状态随机漂移
- 设备 A 点"开始"，设备 B 点"停止" → running 文档并发写
- 跨设备同时点"结束" → P0-B 场景 + 状态漂移

**v9.0.3 P2-2 影响**：v8.2.15 跨设备保护代码虽在，但 `remoteClientId` 永远空字符串，**保护已失效**。

**机制层设计**：
1. 云端定义 `TASK_STATE_MACHINE`（合法转换图）
2. 转换前检查 `expectedState`，不匹配返回 1002（数据冲突）
3. 客户端所有状态转换（start/pause/resume/stop）带 `expectedState`

#### v9.3.0 离线完整化

**核心问题**：v9.1.0 compound + v9.2.0 状态机 + 任务锁都假设"在线"——离线场景下需要特殊处理。

**典型场景**：
- 离线时操作任务（暂停）→ 上线时其他设备已停止该任务 → 状态冲突
- 离线时复合操作入队 → 上线时任务锁申请失败 → 复合操作无法 flush
- 离线时多个操作积累 → 上线时批量冲突 → UX 复杂

**机制层设计**：
1. 离线时本地乐观 UI（已有）
2. 上线时**冲突检测 + 用户决策 UI**（新增）
3. 复合操作离线时**整体入队**（基于 v9.1.0）
4. 任务锁 + 队列的协作：先 flush 锁申请，再处理其他队列项

### 12.4 路线图设计原则

1. **根因导向**：每个版本问"为什么要打这个补丁"——根因 = 缺失的机制层
2. **一次解决一类**：不修单个 bug，修整个机制
3. **已有机制优先启用**：timebankTaskLock 比新设计 state machine 成本低
4. **业务层不手工联动**：业务层代码应该"声明意图"，机制自动保证正确性
5. **机制依赖顺序**：v9.1.0 → v9.2.0 → v9.3.0 不可跳跃

### 12.5 "机制完善"哲学的最终目标

**所有业务路径上的"联动补偿"代码 → 0 行**

```javascript
// 机制完善前的业务层（需要手工联动）
async function completeTask(taskId, transaction) {
    await DAL.addTransaction(transaction);
    try {
        await DAL.stopTask(taskId);
    } catch (e) {
        // 手工联动补偿
        await DAL.deleteTransaction(transaction.id);
        throw e;
    }
}

// 机制完善后的业务层（声明意图即可）
async function completeTask(taskId, transaction) {
    await callCompoundMutation([
        { action: 'addTransaction', data: { ...transaction } },
        { action: 'stopTask', data: { taskId } }
    ], { onRollback: () => { /* 本地 UI 撤销 */ } });
}
```

**"机制完善"的本质**：让业务层代码**从"如何保证正确"变成"声明想要什么"**。
