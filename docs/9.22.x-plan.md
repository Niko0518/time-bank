# 9.22.x 重启计划跟踪

> **状态**：进行中（v9.22.S）
> **创建**：2026-07-21
> **负责人**：开发者本人
> **特殊版本号**：`v9.22.S`（"S" 代表 **S**pecial / **S**econd-chance / **S**afe，纪念 9.22.0 的数据灾难与本次重启）

---

## 1. 背景

### 1.1 为什么重启

9.22.0 曾是一次雄心勃勃的冷启动优化（13s → ~2s），但因为"任务数据精简时删除顶层字段未完整覆盖的 `data` 对象副本"，导致：

- 22 个 `reward` 任务丢失 `fixedTime`
- 4 个 `instant_redeem` 任务丢失 `consumeTime`
- 6 个 `continuous_target` 任务丢失 `targetTime` / `bonusReward`
- 计算产生 `NaN` → `amount=0` → 导出 JSON 时 `null`
- 用户反馈：任务卡片显示 0、完成余额不变、备份出现 `amount: null`

9.22.x 系列（包括 9.22.0 / 9.22.1 / 9.22.4）已整体回退到 9.21.0（见 `docs/restoring-v9.22.0-optimizations.md` 与 `docs/version-changelog.md` v9.21.1 章节）。

### 1.2 重启目标（**质量优先，不追求速度**）

| 维度 | 目标 |
|------|------|
| **数据完整性** | 100% 保持 9.21.0 的数据兼容行为，任何字段都不丢失 |
| **冷启动性能** | 恢复到 9.22.0 同等水平（13s → ~2s）即可，不追求极致 |
| **可回退性** | 单 `git revert HEAD` 即可回退到 9.21.1 |
| **可观测性** | 字段缺失能被立刻发现（打点 + Console） |
| **云函数** | **完全不动**，保留 9.14.1 / 9.12.2 旧版 |

## 2. 核心设计原则

### 2.1 「安全投影」规则

9.22.0 的灾难根因是"**删除重复 data 但顶层字段不全**"。本次重启：

- ✅ 启用 `TASK_PROJECTION` / `TX_PROJECTION`（减少网络包体积）
- ✅ 投影**必须包含** `data: true`（保留所有历史字段副本）
- ❌ **永不动用**"删除 `task.data`"代码
- ✅ 投影字段清单**只用作网络层声明**，不做任何客户端精简

### 2.2 「NaN 兜底」规则

即使有投影保护，仍然在关键计算路径加 `typeof === 'number' && !isNaN` 兜底：

- `processHabitCompletion` 入口（`app-2.js` L4690 附近）
- `processNormalCompletion` 内部（`app-2.js` L4708 附近）
- `addTransaction` 写入前（兜底 amount）

### 2.3 「字段缺失守护」规则

加载完成后，遍历任务对象，若发现"按 type 应有但缺失的顶层字段"，**只打点不上报也不改写**：

- `reward` 缺 `fixedTime`
- `instant_redeem` 缺 `consumeTime`
- `continuous_target` 缺 `targetTime` / `bonusReward`

**理由**：9.22.0 的二次灾难正是"自动改写用户数据"造成的。本次只观测、不干预。

## 6. 禁忌清单（不允许做的事）

- ❌ 任何形式的"自动修复用户数据"（9.22.0 灾难的直接诱因）
- ❌ 删除 `task.data` / `t.data`（即使是"重复"也不删）
- ❌ 投影中省略 `data: true`（会导致老数据兼容问题）
- ❌ 修改云函数（`tbMutation` / `timebankSync` / `timebankAI` / `timebankTaskLock`）
- ❌ 未经"推送"指令执行 `git push`
- ❌ 先卸载再安装 APK
- ❌ 擅自修改版本号（用户已指定 `v9.22.S`，AI 不再拍板）
