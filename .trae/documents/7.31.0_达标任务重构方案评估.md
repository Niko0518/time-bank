# v7.31.0 版本评估报告：达标任务类型重构

## 一、用户需求

将 **达标任务** 从独立任务类型改为：

* 任务创建/编辑面板中的一个**开关**（`hasTarget` / `targetEnabled`）

* **单次任务** 和 **计时任务** 都能设置目标

***

## 二、当前架构分析

### 2.1 任务类型体系

| 类型值                 | 名称       | 说明           |
| ------------------- | -------- | ------------ |
| `reward`            | 按次任务     | 固定奖励         |
| `continuous`        | 计时任务     | 按时长×倍率奖励     |
| `continuous_target` | **达标任务** | 计时+目标时间+达标奖励 |
| `instant_redeem`    | 即时兑换     | 固定消费         |
| `continuous_redeem` | 计时兑换     | 按时长×倍率消费     |

### 2.2 达标任务特有字段

```javascript
{
  type: 'continuous_target',    // 任务类型
  targetTime: 1500,            // 目标时间（秒）
  bonusReward: 300,            // 达标额外奖励（秒）
}
```

### 2.3 达标任务特有运行时状态

```javascript
runningTask = {
  achieved: false,            // 是否已达标
  achievedTime: 0,             // 达标时间戳
}
```

### 2.4 达标任务的 UI/UX 表现

| 场景     | 表现                                 |
| ------ | ---------------------------------- |
| 任务卡片显示 | `目标${formatTime(task.targetTime)}` |
| 计时中徽章  | 显示 `✅` 标记表示达标                      |
| 达标提醒   | 首次达到目标时触发通知                        |
| 结束奖励计算 | 达标时额外奖励 `bonusReward`              |
| 补录/撤销  | 需达标的交易才计入进度                        |

## 四、兼容性挑战

### 4.1 历史数据兼容

**问题**：现有任务的 `type: 'continuous_target'` 如何处理？

**方案**：迁移脚本

* 读取所有任务，将 `type: 'continuous_target'` 改为 `type: 'continuous'` + `hasTarget: true`

* **风险**：用户多端数据可能短暂不一致

### 4.2 习惯系统联动

**问题**：`continuous_target` 与 `habitDetails` 可以组合

```javascript
{
  type: 'continuous_target',
  isHabit: true,
  habitDetails: { period: 'daily', targetCountInPeriod: 3, ... }
}
```

**影响**：

习惯周期内的"次数"达标逻辑（`targetCountInPeriod`）

* 与目标时间达标的**双重达标条件**
* 需要确保改动后习惯系统仍正常工作
* 如果一个单次任务需要完成x次才达标，那么设置为习惯后，周期目标次数只统计周期内该任务达标次数，并非完成次数，且以达标次数计算连胜。

### 4.3 交易记录兼容

**问题**：历史交易的 `type: 'earn'` 如何关联到达标逻辑？

**现状**：交易记录只存储 `taskId` 和 `taskName`，不存储任务类型。验证逻辑通过查询 `tasks` 数组。

**风险**：如果任务被修改（删除 `continuous_target` 类型），历史交易的验证逻辑可能异常。

***

## 五、重构方案设计

### 5.1 推荐方案：渐进式重构（向后兼容）

**核心思路**：

1. **不删除** `continuous_target` 类型值（保持兼容）
2. 新增 `hasTarget: true` 字段表示"启用目标"
3. `continuous` + `hasTarget: true` 等价于原 `continuous_target`
4. UI 层面：任务类型仍显示为"计时任务"，但显示"目标"开关状态

### 5.2 改动点清单

#### 数据层

* [ ] `task.hasTarget`: `boolean` - 是否启用目标（可选，默认为 `false`）

* [ ] `task.targetTime`: 已有字段，继续保留

* [ ] `task.bonusReward`: 已有字段，继续保留

* [ ] `runningTask.achieved`: 已有字段，继续保留

#### 类型判断统一

* [ ] 创建辅助函数 `isTargetEnabled(task)`

  ```javascript
  function isTargetEnabled(task) {
    return task.hasTarget === true || task.type === 'continuous_target';
  }
  ```

* [ ] 替换所有 `task.type === 'continuous_target'` 为 `isTargetEnabled(task)`

* [ ] 替换所有 `['continuous', 'continuous_target']` 为兼容数组

#### UI 层

* [ ] 编辑表单：类型选择器下方增加"目标开关"

* [ ] 目标开关打开时：显示 `targetTime`、`bonusReward` 输入框

* [ ] 目标开关关闭时：隐藏这两个字段（但保留值）

* [ ] 任务卡片显示：计时任务显示"目标"徽章

#### 表单验证

* [ ] `hasTarget === true` 时要求填写 `targetTime`

* [ ] `targetTime > 0` 时 `bonusReward` 可选

#### 习惯联动

* [ ] 习惯任务的周期"次数"达标与目标"时间"达标是**且**关系

* [ ] 需要确保两套逻辑独立工作

***

## 六、实施步骤（渐进式）

### Phase 1: 基础设施（安全）

1. 创建 `isTargetEnabled(task)` 辅助函数
2. 替换核心逻辑中的类型判断（停止、奖励计算）
3. 本地测试确保计时任务功能正常

### Phase 2: UI 适配（功能）

1. 任务卡片显示"目标"徽章
2. 表单增加目标开关
3. 测试创建/编辑带目标的计时任务

### Phase 3: 兼容性确认

1. 现有 `continuous_target` 任务仍正常工作
2. 习惯任务+目标组合仍正常工作
3. 补录/撤销逻辑正常

### Phase 4: 清理（可选）

1. 决定是否保留 `continuous_target` 类型值

***

## 七、风险评估

| 风险       | 等级    | 缓解措施                       |
| -------- | ----- | -------------------------- |
| 现有任务数据损坏 | **高** | 1. 不删除类型值；2. 本地备份；3. 分阶段测试 |
| 习惯系统联动失效 | **中** | 单独测试习惯+目标组合                |
| 交易验证异常   | **中** | 验证逻辑基于 `tasks` 而非 `type`   |
| UI 回归    | **低** | 渐进式改动，每步验证                 |
| 多端数据不一致  | **低** | 云端同步机制已有保护                 |

***

## 八、建议

### 8.1 是否建议现在实施？

**建议：谨慎推进**

* 项目刚经历 v7.30.x 的同步机制重构

* 达标任务与多个系统深度耦合

* 建议：先完成 Phase 1（基础设施），验证无风险后再继续

### 8.2 替代方案：仅优化 UI，不改架构

如果风险过高，可以考虑：

* **保持** **`continuous_target`** **类型不变**

* **仅优化表单 UI**：将"目标"字段从"类型专属"改为"通用字段"

* 这样改动量最小，但架构不变

### 8.3 充分测试建议

* 安卓端完整测试：创建、计时、停止、撤销

* 习惯任务测试：周期内多次计时

* 多端测试：如果有多端的话

* 数据回滚测试：确认本地缓存损坏可以恢复

***

## 九、总结

| 评估项  | 结论             |
| ---- | -------------- |
| 可行性  | ✅ 可行，但需谨慎      |
| 改动规模 | 中等（\~50处代码改动）  |
| 核心风险 | 历史数据兼容性、习惯系统联动 |
| 建议策略 | 渐进式重构，保持向后兼容   |
| 测试要求 | 必须完整回归测试       |

**推荐实施**：采用 Phase 1-3 渐进式，最多延后 Phase 4 清理。
