## 三、云端同步机制

### 3.1 同步触发时机

```
启动 / 重新联网
  ├─ lastCloudSyncAt > 6h 或首次 → activateCloudSyncWriteLock（门禁）
  └─ loadAll() 全量拉取 → 成功后 releaseCloudSyncWriteLock + 更新 lastCloudSyncAt

Watch 重建 / 30s 主动同步触发 reconcileCloudAfterWatch()
  ├─ 距上次同步 < 30 分钟 → fetchDelta() 增量 → mergeTransactionDelta()
  │   └─ fetchDelta 返回 null（云函数未部署）→ 降级全量 loadAll()
  └─ 距上次同步 ≥ 30 分钟 → loadAll() 全量（兜底）

新增交易 DAL.addTransaction()
  ├─ writeTransactionSafe()（云函数幂等写）
  └─ 返回 null（云函数未部署）→ db.collection().add() 直接写入
```

### 3.2 写入门禁机制 (v7.28.0)

**目的**: 防止陈旧端（长时间不活跃）用旧数据覆盖云端新数据

```javascript
// 全局变量
let lastCloudSyncAt = parseInt(localStorage.getItem('tb_lastCloudSyncAt') || '0');
const STALE_SYNC_THRESHOLD_MS = 6 * 60 * 60 * 1000; // 6小时
let cloudSyncWriteLock = false;

// 激活门禁时机
- loadData() 启动时：若 lastCloudSyncAt === 0 或超 6h
- visibilitychange 长休眠后

// 解除门禁时机
- loadAll() 成功完成后
- 15秒超时自动解锁（保险机制）

// 门禁效果
- saveData() 直接 return，不写云端
```

### 3.3 DAL (Data Access Layer) 核心方法

| 方法                             | 用途     | 关键特性                    |
| ------------------------------ | ------ | ----------------------- |
| `DAL.init()`                   | 初始化    | Profile 不存在时自动重建        |
| `DAL.loadAll()`                | 全量加载   | 加载后解除写入门禁               |
| `DAL.loadProfile()`            | 加载用户配置 | 缓存到 `DAL.profileData`   |
| `DAL.saveProfile(data)`        | 保存配置   | 使用 `Object.assign` 更新内存 |
| `DAL.saveTask(task)`           | 保存任务   | 独立同步，不走 saveData        |
| `DAL.fetchDelta(lastSyncAt)`   | 增量拉取   | 云函数实现，失败返回 null         |
| `DAL.writeTransactionSafe(tx)` | 幂等写入   | 防重复写入                   |

## 四、发现的问题

### 问题 1: ⚠️ DAL.saveProfile 架构陷阱（dot-notation key 限制）

**问题描述**:

```javascript
// DAL.saveProfile 内部实现
Object.assign(this.profileData, data);

// 当 data 的 key 是 dot-notation 时
DAL.saveProfile({ "deviceSpecificData.abc": { ... } });
// 结果: profileData.deviceSpecificData["abc"] 在内存中永远不会被更新！
// 只会添加一个 key 为 "deviceSpecificData.abc" 的顶层属性
```

**影响范围**:

* `saveDeviceSpecificData()` / `saveDeviceSpecificDataDebounced()` 都用 dot-notation 写入

* 当前会话内写入的 `autoDetectRawRecords` 在 `DAL.profileData.deviceSpecificData` 中永远不可见

**当前修复状态**:

* ✅ v7.29.2 已修复 `collectAutoDetectRawRecords`：始终优先从本机 localStorage 读取

* ⚠️ 但根本问题（dot-notation 导致内存不同步）仍然存在

**建议修复**:

```javascript
// 修改 DAL.saveProfile 支持嵌套路径
async saveProfile(data) {
    // 检查是否有 dot-notation key
    for (const key in data) {
        if (key.includes('.')) {
            // 使用 _.set 更新嵌套属性
            const path = key.split('.');
            let target = this.profileData;
            for (let i = 0; i < path.length - 1; i++) {
                if (!target[path[i]]) target[path[i]] = {};
                target = target[path[i]];
            }
            target[path[path.length - 1]] = data[key];
        } else {
            Object.assign(this.profileData, { [key]: data[key] });
        }
    }
}
```

### 问题 2: ⚠️ saveData 不保存任务到云端

**问题描述**:

```javascript
// saveData() 只保存 Profile，不保存任务
await DAL.saveProfile({
    reportState: reportState,
    categoryColors: [...categoryColors],
    // ... 不含 tasks
});
```

**影响**:

* 修改任务排序/属性后需单独调用 `DAL.saveTask(task)` 同步到云端

* 容易遗漏，导致多端任务状态不一致

**建议**:

* 在文档中明确标注此行为

* 或考虑在 saveData 中添加任务同步逻辑

***

### 问题 3: ⚠️ 云端守卫依赖全局变量

**问题描述**:

```javascript
// 自动结算函数的云端守卫
if (!hasCompletedFirstCloudSync && isLoggedIn()) {
    console.warn('[functionName] 云端数据尚未加载完成，跳过本次结算');
    return;
}
```

**风险**:

* `hasCompletedFirstCloudSync` 是全局变量，可能在某些边界条件下被错误设置

* 如果 `loadAll()` 失败但 `hasCompletedFirstCloudSync` 仍为 true，可能导致错误结算

**建议**:

* 增加更严格的检查，如同时检查 `lastCloudSyncAt` 时间戳

***

### 问题 4: ⚠️ localStorage 容量风险

**问题描述**:

* 交易记录 `transactions` 可能无限增长

* localStorage 通常只有 5-10MB 容量限制

**当前缓解措施**:

* 云端同步后本地保留完整数据

* 没有自动清理旧交易的机制

**建议**:

* 考虑添加本地数据归档机制

* 或定期提示用户清理旧数据

***

### 问题 5: ⚠️ 多端并发写入冲突

**问题描述**:

* 虽然写入门禁防止了陈旧端覆盖，但活跃端之间的并发写入仍可能冲突

* 例如：两台设备同时修改同一任务

**当前机制**:

* CloudBase Watch 实时监听可以在一定程度上缓解

* 但没有显式的乐观锁或版本控制

**建议**:

* 考虑在关键数据上添加版本号或时间戳检查

## 六、结论与建议

### 6.1 机制总体评价

| 维度     | 评分    | 说明                      |
| ------ | ----- | ----------------------- |
| 数据一致性  | ⭐⭐⭐⭐  | 写入门禁 + Watch 监听基本能保证一致性 |
| 离线支持   | ⭐⭐⭐⭐⭐ | localStorage 缓存完善       |
| 多端同步   | ⭐⭐⭐⭐  | 增量同步 + 全量兜底             |
| 容错性    | ⭐⭐⭐   | 部分边界条件依赖全局变量            |
| 代码可维护性 | ⭐⭐⭐   | dot-notation 问题增加复杂度    |

### 6.2 优先修复建议

1. **高优先级**: 修复 `DAL.saveProfile` 的 dot-notation 问题，确保内存数据一致性
2. **中优先级**: 添加 localStorage 容量监控和预警
3. **低优先级**: 考虑添加更严格的多端并发控制
