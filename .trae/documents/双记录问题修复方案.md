# TimeBank 双记录问题修复方案

## 问题背景

用户反馈完成/结束任务后出现两条完全相同的记录，撤回时行为异常（有时一条撤回两条都被撤回，有时需要撤两条）。

**核心约束**：同步机制改动风险极高，需遵循保守、精准、最小化原则。

---

## 一、问题根因分析

### 根因 1：撤回时云端删除失败导致"记录复活"

**问题链**：
1. 用户撤回记录 → `performLegacyUndo()` 执行
2. 本地立即删除（`transactions.splice`），用户看到"撤回成功"
3. 云端删除异步执行（不等待），网络问题时删除失败
4. 云端记录仍存在 → 后续 Watch 同步时"复活"

**证据位置**：[app-2.js:5484-5493](file:///d:/TimeBank/js/app-2.js#L5484-L5493)

### 根因 2：addTransaction 云端写入失败后重试与 Watch 同步的竞态

**问题链**：
1. `addTransaction(tx)` 写入云端，网络抖动
2. 云端写入失败（pending状态），本地已有记录
3. Watch 先收到其他设备事件，触发重新同步
4. 或者重试成功后，Watch 事件与本地操作产生竞争

**证据位置**：[app-auth.js:2598-2672](file:///d:/TimeBank/js/app-auth.js#L2598-L2672)

### 根因 3：去重机制过于宽松（1秒窗口）

**问题链**：
1. 双设备几乎同时操作同一任务
2. 去重条件：`clientId + taskId + amount + 1秒内`
3. 不同 clientId 的操作无法被去重检测

**证据位置**：[app-1.js:3371-3382](file:///d:/TimeBank/js/app-1.js#L3371-L3382)

### 根因 4：任务完成/结束与云端同步的竞态

**问题链**：
1. 用户完成任务 → `processNormalCompletion()` 添加交易
2. 交易已添加到本地 `transactions` 数组
3. 云端写入尚未完成时，Watch 收到变更事件
4. 本地缓存尚未更新，Watch 逻辑认为需要添加

---

## 二、修复策略

### 核心原则
1. **最小改动**：只改必要的地方，不重构同步机制
2. **防御性编程**：增加必要检查，但不过度保护
3. **保守去重**：收紧去重条件，但留有合理窗口
4. **同步等待**：关键操作等待云端确认后再更新UI

---

## 三、具体修复方案

### 修复 1：撤回操作改为同步等待（最小化改动）

**目标**：确保撤回成功后再更新UI，防止"假撤回"

**改动位置**：`app-2.js` 的 `performLegacyUndo` 函数

**改动内容**：
```javascript
// 原代码（异步，不等待）
if (isLoggedIn()) {
    DAL.deleteTransaction(transaction.id)
        .then(() => { ... })
        .catch(err => { ... });  // 不阻塞
}

// 修改为（同步等待）
if (isLoggedIn()) {
    try {
        await DAL.deleteTransaction(transaction.id);
        console.log('[undoTransaction] ✅ 云端删除成功:', transaction.id);
    } catch (err) {
        console.error('[undoTransaction] ❌ 云端删除失败:', err.code, err.message);
        // 云端删除失败时，不执行本地删除，回滚操作
        // 恢复余额
        if (transaction.type === 'earn') {
            currentBalance += transaction.amount;
            updateDailyChanges('earned', transaction.amount, new Date(transaction.timestamp));
        } else {
            currentBalance -= transaction.amount;
            updateDailyChanges('spent', transaction.amount, new Date(transaction.timestamp));
        }
        // 恢复 completionCount
        if (task && task.completionCount !== undefined) {
            task.completionCount++;
        }
        showNotification('⚠️ 撤回失败', `云端删除失败: ${err.message}，本地未撤回`, 'error');
        return; // 不继续执行本地删除
    }
}
```

**风险评估**：低 - 只影响撤回操作，不影响其他同步流程

### 修复 2：撤回前增加本地状态检查（防止重复删除）

**目标**：防止因缓存问题导致的重复删除

**改动位置**：`app-2.js` 的 `performLegacyUndo` 函数

```javascript
function performLegacyUndo(transaction, transactionIndex, task) {

    const existingTx = transactions.find(t => t.id === transaction.id);
    if (!existingTx) {
        console.warn('[performLegacyUndo] 记录已不存在，跳过:', transaction.id);
        return;
    }

    // ... 原有逻辑 ...
}
```

**风险评估**：极低 - 只增加防御性检查

---

### 修复 3：收紧去重时间窗口（从1秒降至500ms）

**目标**：减少并发操作产生重复记录的可能

**改动位置**：`app-1.js` 第 3382 行

**改动内容**：
```javascript
// 原代码
Math.abs(t.timestamp - txTimestamp) <= 1000  // 1秒

// 修改为
Math.abs(t.timestamp - txTimestamp) <= 500  // 500ms，更严格但仍有合理窗口
```

**风险评估**：低 - 只影响去重判断逻辑，不影响同步流程

---

### 修复 4：增加本地操作完成的标记机制（解决竞态）

**目标**：防止 Watch 事件在本地操作完成前触发同步

**改动位置**：
- `app-reports.js` 的 `addTransaction` 函数开头
- `app-1.js` 的 Watch 处理逻辑

**改动内容**：

在 `addTransaction` 开头增加标记：
```javascript
function addTransaction(transaction) {
    // 新增：标记为最近本地操作，Watch 忽略此期间的同名操作
    const isLocalOperation = !transaction._fromCloud;
    if (isLocalOperation && transaction.id) {
        recentLocalTransactions.set(transaction.id, Date.now());
    }
    // ... 原有逻辑 ...
}
```

在 Watch 的 add 处理中增加检查：
```javascript
// 在检测重复之前，增加对本端操作的忽略
if (txClientId === clientId) {
    const localAddTime = recentLocalTransactions.get(txId);
    if (localAddTime && (Date.now() - localAddTime) < 2000) {
        console.log(`🛡️ [Watch] 忽略本端触发的 add（1）: ${txId}`);
        this.transactionCache.set(txId, doc._id || doc.id);
        continue;
    }
}
```

**风险评估**：中 - 增加了状态管理，但只用于临时忽略

---

### 修复 5：stopTask/completeTask 完成后增加短暂保护期

**目标**：确保任务完成后，数据已同步后再允许其他操作

**改动位置**：`app-2.js` 的 `completeTask` 和 `stopTask` 函数

**改动内容**：
```javascript
async function stopTask(taskId) {
    // ... 原有逻辑 ...

    // 在 saveData 之前设置保护标记
    lastLocalActionTime = Date.now();

    await saveData();

    // 保护期结束后（200ms），再更新一次 UI 确保显示正确
    setTimeout(() => {
        updateAllUI();
    }, 200);
}
```

**注意**：这只是一个 UX 优化，确保最终一致性。

---

### 修复 6：addTransaction 返回 Promise 供调用方等待

**目标**：关键操作完成后等待云端同步

**改动位置**：`app-reports.js` 的 `addTransaction` 函数

**现状**：`addTransaction` 已经返回 Promise（v7.32.0-fix），但调用方大多没有 await

**改动内容**：
```javascript
// 在 processNormalCompletion 中 await addTransaction
async function processNormalCompletion(...) {
    // ... 计算逻辑 ...

    await addTransaction(transaction);  // 新增 await

    // 后续逻辑依赖交易已添加完成
    currentBalance += adjustedTime;
    task.completionCount = (task.completionCount || 0) + 1;
    updateDailyChanges('earned', adjustedTime, referenceDate);
    // ...
}
```

**风险评估**：中 - 需要确保所有调用方都能正确处理 Promise

## 四、不推荐的改动（过度设计）

以下改动**不做**，因为过于复杂或风险过高：

1. ~~重构整个同步机制~~ - 风险太高
2. ~~引入事务/锁机制~~ - 可能造成死锁或性能问题
3. ~~改变 Watch 处理逻辑~~ - 可能引发其他同步问题
4. ~~增加版本向量/向量时钟~~ - 过度工程化
5. ~~修改 saveData 的写入逻辑~~ - 核心机制，牵一发动全身

## 五、修复优先级排序

| 优先级 | 修复项 | 改动范围 | 风险 | 预期效果 |
|--------|--------|----------|------|----------|
| P0 | 修复1：撤回同步等待 | app-2.js | 低 | 防止撤回假成功 |
| P0 | 修复2：撤回前检查 | app-2.js | 极低 | 防止重复删除 |
| P1 | 修复3：收紧去重窗口 | app-1.js | 低 | 减少并发重复 |
| P2 | 修复4：本地操作标记 | app-reports.js + app-1.js | 中 | 减少竞态 |
| P3 | 修复6：await addTransaction | app-2.js | 中 | 确保顺序执行 |

**P0 是必须修复的，P1-P3 根据情况决定是否实施。**

## 六、测试建议

修复后需要验证：
1. 单设备快速完成/撤回 10 次，观察是否有重复
2. 双设备同时操作同一任务，观察数据一致性
3. 网络不稳定时操作，观察是否有假撤回/假成功
4. 观察控制台是否有异常日志

---

## 七、Rollback 计划

如果修复后出现问题，可以：
1. 修复1：改回异步即可
2. 修复2：删除检查代码即可
3. 修复3：改回 1000ms 即可
4. 修复4：删除标记逻辑和检查逻辑即可
5. 修复6：移除 await 即可

所有修复都是**可逆的**，不会破坏现有结构。
